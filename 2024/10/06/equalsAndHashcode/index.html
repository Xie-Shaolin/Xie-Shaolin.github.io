    1. 重写了equals方法为什么又要重写hashcode方法

    重写了equals方法为什么又要重写hashcode方法

    首先这是Object类对hashCode()方法的契约（contract）的一部分，具体来说是：如果两个对象根据equals(Object o)方法是相等的，那么调用这两个对象中任一对象的hashCode()方法都必须产生相同的整数结果。

    为什么会有这个要求？

    这主要是为了确保对象在使用基于哈希的集合（如HashSet、HashMap、Hashtable和LinkedHashSet等）时能够正确地工作。

    在讲解这个之前，我们需要简单说明一下HashMap的工作原理：

    • HashMap map = new HashMap(): 在实例化以后，底层创建了长度是16的一维数组Entry[] table。
    • map.put(key1,value1)：调用key1所在类的hashCode()计算key1哈希值，此哈希值经过某种算法计算以后，得到在Entry数组中的存放位置。
      • 如果此位置上的数据为空，此时的key1-value1添加成功
      • 如果此位置上的数据不为空，(意味着此位置上存在一个或多个数据(以链表形式存在)),比较key1和已经存在的一个或多个数据的哈希值：
        • 如果key1的哈希值与已经存在的数据的哈希值都不相同，此时key1-value1添加成功。—-情况2（以链表的方式存储）
        • 如果key1的哈希值和已经存在的某一个数据(key2-value2)的哈希值相同，继续比较：调用key1所在类的equals(key2)方法，比较：
          • 如果equals()返回false:此时key1-value1添加成功。—-情况3 （以链表的方式存储）
          • 如果equals()返回true:使用value1替换value2
    • 在不断的添加过程中，会涉及到扩容问题，当超出临界值(且要存放的位置非空)时，扩容。默认的扩容方式：扩容为原来容量的2倍，并将原有的数据复制过来。
    • jdk7底层结构只有：数组+链表。jdk8中底层结构：数组+链表+红黑树。
    • 当数组的某一个索引位置上的元素以链表形式存在的数据个数>8 且当前数组的长度>64时，此时此索引位置上的所数据改为使用红黑树存储
    • map.get(key1)：
      • 调用的key1的 hashCode() 方法来计算其哈希码
      • 通过哈希码计算出一个数组索引
      • 一旦计算出了索引位置，HashMap 就会遍历该位置上的链表或红黑树。
        • 对于链表，它会逐个比较节点的键是否与传入的键相等（使用 equals() 方法）。
        • 对于红黑树，它会利用红黑树的特性来高效地搜索键。
      • 如果找到了匹配的键，则返回对应的值；否则，返回 null

    如果不遵守这个约定，相等的对象有不同的哈希码，那么它们可能会被放置在不同的桶中，导致哈希表查找、插入和删除操作的性能下降，甚至在最坏的情况下退化为链表（链表冲突）。

    • 比如，我们要创建一个key为User，value为Order的HashMap，如果相同内容User（同一个用户）计算出不同的hashcode
    • 我原本要去覆盖之前的值，现在会变成新增一个值，也就是同一个key会产生多个值。如果我们想取出某个对象的时候，我们也可能取不出来。

    拓展问题： 两个对象的hashCode()相同，则equals()是否也一定为 true ?

    两个对象equals相等，则它们的hashcode必须相等，如果两个对象的 hashCode()相同，则equals()不一定为true。这是因为哈希码值的范围（int类型，即-2^31^到2^31-1^）是有限的，而对象可以是无限多的，因此不同的对象有可能具有相同的哈希码值（哈希冲突）。

    如果两个对象通过equals()方法比较是相等的，那么它们应该具有相同的哈希码值。这是Object类对hashCode()方法的契约（contract）的一部分，具体来说是：如果两个对象根据equals(Object o)方法是相等的，那么调用这两个对象中任一对象的hashCode()方法都必须产生相同的整数结果。
